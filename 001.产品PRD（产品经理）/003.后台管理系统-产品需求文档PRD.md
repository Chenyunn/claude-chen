## 一、产品概述

### 1.1 产品背景

「每日记账」App 已上线注册登录、记账、统计、导出等 C 端功能，用户与账目数据持续积累。为支持运营专员日常开展**用户运营、数据复盘、风险管控**，需要一套独立的「后台管理系统」，对平台用户、账目流水、分类字典、导出任务等进行集中查看与管理。

### 1.2 产品定位

面向**内部运营专员**使用的 Web 端管理后台，与 C 端 App 共用同一套数据库（只读为主、受限写操作），提供数据概览、用户管理、账目核查、统计分析等能力。

### 1.3 目标用户

- **运营专员**：查看用户增长、处理异常账户、复盘平台收支数据
- **客服/支持**：根据用户名排查用户账目问题、协助重置密码/解封
- （后续可扩展）**数据分析**角色

### 1.4 设计原则

- **只读优先**：绝大多数页面为查询/统计，不直接干预业务数据
- **最小写权限**：必要的写操作（封禁、解封、强制下线、重置密码）仅更新**已有字段**
- **最小建表**：除新增一张「超级管理员表」外，不新增其它表、不修改任何已有表结构
- **可追溯**：写操作记录到应用层日志（见第六章约束说明）

## 二、核心约束（重要）

> **前提要求：除新增一张「超级管理员表」外，禁止修改数据库及其它表结构。**
> - ✅ 允许：新增**唯一一张** `admin_users` 表，**仅用于**超级管理员/管理员账号与登录。
> - 🚫 禁止：修改任何已有表（`users`、`transactions` 等）的结构，也不得新增除此以外的其它表。

本系统功能须在**现有 `daily_expense` 表结构 + 一张新增 `admin_users` 表**之上实现。下表列出运营常见诉求与可行性结论：

| 运营运诉求 | 是否可行 | 实现方式（不改表） |
|-----------|---------|------------------|
| 查看/搜索用户 | ✅ 可行 | 读 `users` 表 |
| 封禁 / 解封用户 | ✅ 可行 | 更新 `users.status`（1正常 ↔ 2锁定 ↔ 9禁用） |
| 重置用户密码 | ✅ 可行 | 更新 `users.password_hash`、清零 `failed_login_count` |
| 强制下线（踢人） | ✅ 可行 | 更新 `auth_tokens.revoked_at` |
| 删除/隐藏某笔账目 | ✅ 可行 | 更新 `transactions.deleted_at`（软删除） |
| 启用/停用分类 | ✅ 可行 | 更新 `categories.is_active` |
| 超级管理员账号 / 登录 | ✅ 可行 | 新增**唯一一张** `admin_users` 表承载（见 6.1） |
| 操作审计日志 | ⚠️ 受限 | 新表名额已用于管理员账号 → 仍走**应用层文件日志**（见 6.2） |

**结论**：得益于现有表已普遍设计 `status`、`deleted_at`、`revoked_at`、`is_active` 等字段，**绝大多数运营操作无需改表即可实现**；管理员登录通过**新增一张 `admin_users` 表**解决；仅"操作审计日志"仍需走应用层文件日志（见 6.2）。

## 三、数据基础

后台功能基于以下表与视图（其中 `admin_users` 为本期**唯一新增表**，其余均为已有表；**只读或仅更新已有列**）：

|--------|------|---------|

| `admin_users` ✨新增 | 超级管理员账号 | 管理员登录、账号管理（唯一新增表） |
| `user_settings` | 用户设置 | 查看用户偏好（币种、提醒、深色模式等） |

| `categories` | 主分类 | 分类字典管理、启用/停用 |
| `category_subcategories` | 子分类 | 子分类查看 |
| `payment_account_types` | 支付账户类型字典 | 字典查看 |
| `payment_accounts` | 用户支付账户 | 用户账户查看 |

| `export_tasks` | 导出任务 | 导出任务监控、清理 |

## 六、约束专项说明

### 6.1 超级管理员账号（新增唯一一张表）

经需求确认，允许**新增唯一一张** `admin_users` 表，**仅用于**承载超级管理员/管理员账号与登录，与 C 端 `users` 表完全隔离。最终建表脚本由 DBA 落库到 `004.数据库脚本` 目录。

**建议表结构（DDL 草案）：**

```sql
CREATE TABLE admin_users (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '管理员ID',
  username VARCHAR(50) NOT NULL COMMENT '登录用户名',
  password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希，不保存明文',
  nickname VARCHAR(50) NULL COMMENT '昵称',
  role ENUM('super_admin','admin') NOT NULL DEFAULT 'admin' COMMENT '角色：super_admin 超级管理员 / admin 普通管理员',
  status TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '状态：1正常，2锁定，9禁用',
  failed_login_count INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '连续登录失败次数',
  locked_until DATETIME(3) NULL COMMENT '锁定截止时间',
  last_login_at DATETIME(3) NULL COMMENT '最近登录时间',
  last_login_ip VARCHAR(45) NULL COMMENT '最近登录IP',
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
  PRIMARY KEY (id),
  UNIQUE KEY uk_admin_users_username (username),
  KEY idx_admin_users_status (status),
  CONSTRAINT chk_admin_users_status CHECK (status IN (1, 2, 9))
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='后台管理员账号表（后台系统唯一新增表）';
```

**设计要点：**

- **与 `users` 隔离**：管理员账号不进 C 端 `users` 表，避免污染用户数据，也无需给 `users` 加 role 字段。
- **角色区分**：`role` 区分 `super_admin`（可管理其它管理员账号，见 1.5）与 `admin`（仅日常运营）。
- **会话方式**：管理员登录签发 **JWT（无状态）**，不占用 `auth_tokens` 表（其 `user_id` 外键指向 `users`，无法承载管理员）；登出在前端丢弃 Token，服务端可选用内存黑名单做强制失效。
- **初始化**：落库时由 DBA 插入一个 `super_admin` 种子账号（密码哈希），首次登录后修改。

### 6.2 操作审计日志（仍走应用层）

新增表的"一个名额"已用于 `admin_users`，**审计日志不再新增表**，继续走应用层：

- 所有管理端写操作记录写入**应用层日志文件**（如 `logs/admin-audit.log`），包含：操作管理员、操作时间、目标用户/账目、操作类型、前后值。
- 后台"操作日志查看"页直接读取/检索该日志文件。

> 后续如需结构化审计，可在另行评估后再新增 `admin_operation_logs` 表（本期不做，受"仅一张新表"约束）。

### 10.2 名词解释

| 术语 | 说明 |
|------|------|
| 后台管理系统 | 面向内部运营专员的 Web 管理端，与 C 端 App 共用数据库 |
