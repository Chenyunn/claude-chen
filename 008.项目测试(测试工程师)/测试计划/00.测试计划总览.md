## 一、被测系统概述

「每日记账」是一款个人日常记账应用，由三端代码共同构成同一产品：

| 端 | 代码位置 | 形态 | 角色 |
| --- | --- | --- | --- |
| 后端服务 | `003.前端代码（前端开发工程师）/backend` | Express + MySQL 真实服务 | 提供 `/api/*` 接口，数据与鉴权中枢 |
| 前端 Web | `003.前端代码（前端开发工程师）/frontend` | 原生 HTML/CSS/JS 多页应用（MPA） | PC/浏览器端入口 |
| 跨端 APP | `007.跨端APP应用(移动端开发工程师)/uniapp-projct` | uni-app (Vue3 + TS) | H5 / 小程序 / App 多端入口 |

### ⚠️ 整体关键结论（影响测试策略）

1. **三端均为半成品**：数据库设计了 9 张完整业务表，但后端**仅实现了「鉴权」和「分类管理」两个模块**；记账流水、支付账户、统计、导出等核心功能**未落地**。前端与 uni-app 中对应的记账/交易/统计页面多为**静态原型**或**未接 API**。
2. **端到端真正打通的业务仅**：注册 / 登录 / 登出 / 获取用户信息 / 分类 CRUD。
3. 因此本轮测试的**核心目标**是：① 保证已实现功能的正确性与安全；② 把半成品与原型页的边界讲清楚，避免误测；③ 充分暴露已识别的高风险缺陷。

## 二、测试范围

### 2.1 总体范围（In Scope）

| 范围 | 说明 |
| --- | --- |
| 功能测试 | 已实现模块（鉴权、分类管理）的端到端功能；各端 UI 交互与静态页可用性 |
| 接口测试 | 后端 `/api/*` 接口的契约、参数校验、鉴权、异常 |
| 兼容性测试 | 浏览器端多浏览器/分辨率；uni-app 的 H5/微信小程序/App 跨端 |
| 安全专项测试 | 鉴权、Token、密码、CORS、注入、敏感信息泄露 |
| 静态质量 | 类型检查、Lint、依赖安全、配置与硬编码 |

### 2.2 暂不纳入范围（Out of Scope）

| 排除项 | 原因 |
| --- | --- |
| 记账流水 / 支付账户 / 统计 / 导出的后端逻辑 | 后端尚未实现，无接口可测（仅在 UI 层做「原型可用性」检查） |
| 真实支付 / 订单 / 第三方支付回调 | 产品未集成真实支付，"支付账户"仅为记账标签 |
| 微信/手机号登录、短信验证码 | 产品仅支持账号密码登录，无此类功能 |
| 高并发/大流量性能压测 | 个人记账类应用、无明确性能指标，仅做轻量并发验证 |
| 上线/生产部署验收 | 当前为开发期，部署配置（弱密钥、root 账号、CORS）问题以缺陷形式提交，不做生产验收 |

> 说明：当后端补齐记账/统计等模块后，需**新增对应测试计划并回归本总览**。

## 三、测试分层模型（统一约定）

本项目统一采用「**倒金字塔 + 专项**」的分层策略：因三端均无既有测试框架、且为半成品，本轮以**接口测试 + 手工功能/兼容性测试 + 安全专项**为主体，辅以静态检查与轻量单测。

```
                    ┌──────────────────────────┐
   L5  端到端(E2E)  │  三端联调：浏览器/Web+后端 │   手工为主
                    ├──────────────────────────┤
   L4  兼容性/专项  │  浏览器兼容 / uni-app 跨端 / 安全 / 性能 │  ★本轮重点
                    ├──────────────────────────┤
   L3  功能/UI 测试 │  各页面/模块功能用例（手工+脚本） │  ★本轮重点
                    ├──────────────────────────┤
   L2  接口测试     │  后端 /api 契约+鉴权+异常（Supertest）│  ★后端重点
                    ├──────────────────────────┤
   L1  单元测试     │  validators / utils / store 纯逻辑   │  轻量补齐
                    ├──────────────────────────┤
   L0  静态检查     │  type-check / Lint / npm audit / 配置 │  前置门槛
                    └──────────────────────────┘
```

各端分层侧重不同，详见各子计划。

## 四、测试环境

### 4.1 后端依赖（**所有端测试的前置条件**）

| 项 | 要求 |
| --- | --- |
| Node.js | 建议 16+（无 engines 声明；@hapi/address 最低 14） |
| MySQL | 9.x（执行 `004.数据库脚本（数据库管理员DBA）/001_daily_expense_schema.sql` 初始化） |
| 后端端口 | `localhost:8081`（注意：前端代码 baseURL 指向 8081，而源码注释写 8080，**以 8081 为准**） |
| API 前缀 | `/api`（如 `http://localhost:8081/api/auth/login`） |
| 数据库账号 | `.env` 配置 root/root（仅测试环境） |

> **联调前提**：进行前端/uni-app 功能测试前，必须先启动 MySQL + 后端服务，否则登录/分类等功能不可用。

### 4.2 各端运行环境

| 端 | 启动方式 | 测试工具 |
| --- | --- | --- |
| 后端 | `npm run dev`（nodemon） | Postman / curl / Supertest 脚本 |
| 前端 Web | 任意静态服务器（Live Server / `python -m http.server`）打开 `index.html` | 浏览器 DevTools / 多浏览器 |
| uni-app H5 | `npm run dev:h5`（端口 3000） | 浏览器 |
| uni-app 微信小程序 | `npm run dev:mp-weixin` + 微信开发者工具 | 微信开发者工具 |
| uni-app App | `npm run dev:app` + HBuilderX/真机 | 真机 / 模拟器 |

> ⚠️ uni-app `manifest.json` 中 `mp-weixin.appid` 为空，微信小程序仅可在**开发者工具本地预览**，无法上传/真机审核测试。

## 八、总体测试风险

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 三端均为半成品，业务边界不清 | 易把原型页当功能测，浪费工时并产生无效缺陷 | 测试前与产品/开发确认「已实现 vs 原型」清单；原型页仅做可用性检查 |
| 接口地址硬编码 `localhost:8081`（http） | 联调与环境切换易出错 | 测试环境统一使用 8081；记录为配置缺陷 |
| 后端 `.env` 含真实密钥/root 账号入库 | 安全与环境污染 | 测试库独立、密码不强相关；登记安全缺陷 |
| uni-app 跨端兼容问题密集 | 小程序/App 端可能大面积不可用 | 优先 H5 端功能验证，跨端做矩阵记录，兼容缺陷集中评估 |
| 无既有自动化测试框架 | 回归成本高、效率低 | 本轮以手工为主；同步搭建接口自动化骨架供后续回归 |
