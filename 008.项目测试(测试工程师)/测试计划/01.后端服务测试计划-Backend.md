## 一、项目概述

### 1.1 技术栈

| 维度 | 选型 |
| --- | --- |
| 语言 | 纯 JavaScript（非 TS） |
| Web 框架 | Express 4.21.0 |
| 数据库 | **MySQL 9.x（真实库，非 mock）**，驱动 mysql2 ^3.11.0，连接池 + Promise API，**无 ORM，全部手写参数化 SQL** |
| 缓存 / MQ | 无 |
| 认证 | JWT（jsonwebtoken ^9.0.2）+ Bearer Token；`auth_tokens` 表做会话撤销记录 |
| 密码加密 | bcryptjs ^2.4.3（cost = 12） |
| 参数校验 | Joi ^18.2.3（仅鉴权模块）；express-validator 已装但未用 |
| 跨域 | cors ^2.8.5（dev 环境全域放开 + credentials） |
| 配置 | dotenv ^16.4.5 |
| 测试框架 | **无（未配置 jest/mocha，无 test 脚本）** |

### 1.2 目录结构

```
backend/
├── .env / .env.example
├── package.json
└── src/
    ├── server.js          # 启动入口（监听端口）
    ├── app.js             # Express 装配（中间件/路由/错误处理/健康检查）
    ├── config.js          # 配置加载
    ├── db.js              # MySQL 连接池
    ├── routes/            # auth.routes.js / category.routes.js
    ├── controllers/       # auth.controller.js / category.controller.js（含 SQL）
    ├── validators/        # auth.validator.js（Joi）
    └── middleware/        # authenticate.js / validate.js / errorHandler.js
```

分层为 **route → controller → pool.query**，无独立 service/dao/model 层。

### 1.3 启动方式

| 命令 | 作用 |
| --- | --- |
| `npm run dev` | nodemon 热重载开发 |
| `npm start` | 生产启动 |
| （无）build / test | 纯 JS 无构建；测试需自建 |

- 端口：`PORT`（`.env` 默认配置，**前端 baseURL 指向 8081，以 8081 为准**）
- 数据库初始化：手动执行 `004.数据库脚本（数据库管理员DBA）/001_daily_expense_schema.sql`
- 必要配置：`DB_*`、`JWT_SECRET`、`JWT_EXPIRES_IN=24h`、`JWT_REMEMBER_EXPIRES_IN=7d`、`LOGIN_MAX_ATTEMPTS=5`、`LOGIN_LOCK_MINUTES=15`、`CLIENT_URL`

## 二、测试目标与范围

### 2.1 目标

1. 验证已实现接口（鉴权、分类管理）的**功能正确性**与**契约一致性**。
2. 验证**鉴权体系**的健壮性（注册/登录/登出/Token/锁定）。
3. 暴露已识别的**高危安全缺陷**并形成可复现用例。
4. 验证**数据完整性**（参数校验、事务、软删除）。

### 2.2 范围内接口

| Method | Path | 鉴权 | 说明 |
| --- | --- | :---: | --- |
| GET | `/api/health` | 否 | 健康检查 |
| POST | `/api/auth/register` | 否 | 注册 |
| POST | `/api/auth/login` | 否 | 登录 |
| POST | `/api/auth/logout` | 是 | 登出 |
| GET | `/api/auth/me` | 是 | 获取当前用户 |
| GET | `/api/categories` | 是 | 分类列表（支持 `?type=`） |
| GET | `/api/categories/:id/subcategories` | 是 | 子分类 |
| POST | `/api/categories` | 是 | 创建分类 |
| PATCH | `/api/categories/:id` | 是 | 修改分类 |
| DELETE | `/api/categories/:id` | 是 | 软删除分类 |

### 2.3 范围外

- 记账流水 / 支付账户 / 统计 / 导出相关接口（**后端未实现**）。
- 真实支付、第三方回调。

## 三、测试分层方案

| 层级 | 内容 | 工具/方式 | 本轮优先级 |
| --- | --- | --- | --- |
| **L0 静态检查** | ESLint（若引入）、`npm audit` 依赖漏洞、`.env` 泄露检查、硬编码扫描 | npm audit / 手工 | ★ 必须 |
| **L1 单元测试** | Joi 校验 Schema（auth.validator）、config 加载、密码哈希边界 | Jest（建议引入） | ☆ 建议 |
| **L2 接口/集成测试** | 全量 `/api` 接口：正常 + 校验 + 鉴权 + 异常 + 事务 | **Supertest + Jest + 测试库** | ★★★ 核心 |
| **L3 数据层验证** | 软删除、外键、事务回滚、并发登录锁定竞态 | 直连 MySQL 校验数据 | ★★ 重要 |
| **L4 安全专项** | Token 撤销、SQL 注入、弱密钥、CORS、bcrypt、锁定爆破 | 手工 + 脚本 | ★★★ 核心 |
| **L5 联调 E2E** | 与前端/uni-app 联调 | 手工 | ★★ 重要 |

### 3.1 测试数据策略

- 使用**独立测试库**（如 `daily_expense_test`），每个接口用例套件 `beforeEach` 清理相关表，保证用例隔离与可重复。
- schema 复用 `001_daily_expense_schema.sql`。
- 不复用开发库，避免 root/root 弱口令污染。

## 四、接口测试用例矩阵（L2/L3）

### 4.1 注册 `POST /api/auth/register`

| 编号 | 场景 | 预期 | 优先级 |
| --- | --- | --- | :---: |
| RG-01 | 合法用户名(6-20字母数字)+密码(6-20)+勾选协议 | 201，返回 token，users/user_settings/auth_tokens 各写入记录 | P0 |
| RG-02 | 用户名 <6 / >20 / 含特殊字符 | 400，Joi 校验失败 | P0 |
| RG-03 | 密码 <6 / >20 | 400 | P0 |
| RG-04 | 用户名已存在 | 409（冲突） | P0 |
| RG-05 | 缺少必填字段 | 400 | P1 |
| RG-06 | 注册事务中断（模拟 user_settings 插入失败） | 整体回滚，不留脏数据 | P1 |
| RG-07 | 密码入库为 bcrypt 哈希（非明文） | 直连库校验 password 字段为哈希 | P0 |

### 4.2 登录 `POST /api/auth/login`

| 编号 | 场景 | 预期 | 优先级 |
| --- | --- | --- | :---: |
| LG-01 | 正确账号密码 | 200，返回 token，过期时间符合 24h/7d(rememberMe) | P0 |
| LG-02 | 密码错误 | 401，失败计数 +1 | P0 |
| LG-03 | 连续 5 次失败 | 账号锁定 15 分钟，期间正确密码也拒绝 | P0 |
| LG-04 | 锁定期满后正确密码 | 登录成功，计数清零 | P0 |
| LG-05 | rememberMe=true | token 过期时间为 7d | P1 |
| LG-06 | 登录成功后失败计数清零 | 直连库校验 | P1 |
| LG-07 | 并发场景：同一账号多次错误密码同时请求 | 失败计数正确（**已知：login 多次 UPDATE 未包事务，可能不一致**） | P1 |

### 4.3 登出 `POST /api/auth/logout`（★高危）

| 编号 | 场景 | 预期 | 优先级 |
| --- | --- | --- | :---: |
| LO-01 | 登出后 `auth_tokens` 该用户记录被撤销 | 直连库校验 | P0 |
| LO-02 | **登出后旧 token 仍可访问 `/api/auth/me`** | **预期应 401；若仍 200 则为 P0 缺陷**（已知：authenticate 未校验 auth_tokens 撤销） | **P0** |
| LO-03 | 无 token / 无效 token 调用 logout | 401 | P1 |
| LO-04 | 登出不影响其他设备 token（若设计为撤销该用户全部则全部失效） | 依设计核对 | P1 |

### 4.4 获取用户 `GET /api/auth/me`

| 编号 | 场景 | 预期 | 优先级 |
| --- | --- | --- | :---: |
| ME-01 | 携带合法 token | 200，返回用户信息 + settings | P0 |
| ME-02 | 无 token | 401 | P0 |
| ME-03 | 伪造/过期 token | 401 | P0 |
| ME-04 | token 签名错误（弱密钥场景） | 401 | P1 |

### 4.5 分类管理 `/api/categories`（均需鉴权）

| 编号 | 场景 | 预期 | 优先级 |

| CAT-02 | GET `?type=expense/income` 过滤 | 仅返回匹配分类 | P1 |
| CAT-03 | GET 子分类 `:id/subcategories` | 200 | P1 |

## 五、安全专项测试（L4）★本轮核心

| 编号 | 专项 | 测试方法 | 预期/已知缺陷 | 优先级 |
| --- | --- | --- | --- | :---: |
| SEC-01 | **登出 Token 撤销失效** | 登出后用旧 token 调 `/api/auth/me` | 预期 401；**已知会返回 200（authenticate.js:34-37 不查 auth_tokens）→ P0** | P0 |
| SEC-02 | **`.env` 泄露** | 检查仓库是否提交 `.env`（含 JWT_SECRET/DB_PASSWORD=root） | 已知已入库 → P0 | P0 |
| SEC-03 | **JWT 弱密钥兜底** | 清空 `JWT_SECRET` 启动，用 `default_secret_change_me` 自签 token 访问 | 预期拒绝；**已知 config.js:30 有弱兜底 → P1** | P1 |
| SEC-04 | **DB root/root 弱口令** | 用 root/root 直连 MySQL | 已知 → P1（仅测试环境） | P1 |
| SEC-05 | **CORS dev 全域+凭证** | 未设 NODE_ENV 时任意 Origin 携带 cookie 访问 | 已知默认 development 全开 → P2 | P2 |
| SEC-06 | **SQL 注入** | 对 username/code/name/name 等注入 payload | 参数化查询应全部安全（已知防护良好） | — |
| SEC-07 | **分类模块缺校验** | POST/PATCH 传入超长、非法类型、负数 sortOrder | 已知无 Joi，仅非空判断 → P2（数据完整性） | P2 |
| SEC-08 | **密码强度** | 弱密码（123456）能否注册 | 依设计核对（当前仅长度校验） | P2 |
| SEC-09 | **接口频率/爆破** | 登录接口高频请求 | 有 5 次锁定机制；建议补 IP 级限流 | P2 |
| SEC-10 | **敏感日志** | 检查日志是否打印密码/token 明文 | 已知仅打印 method+path，风险低 | P3 |
